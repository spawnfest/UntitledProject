What's the big deal ?

When your browser talks to to a web server, it uses one of several "HTTP" verbs, (GET PUT POST PATCH DELETE, ETC). The HTML standard (even HTML 5) has no way of allowing hypertext to make these basic types of requests.

I've listed below 4 of the most common types, which you can roughly map to basic operations in a database.

HTTP Verb Description
GET The GET method requests a representation of the specified resource. Requests using GET should only retrieve data.
POST The POST method submits an entity to the specified resource, often causing a change in state or side effects on the server.
PUT The PUT method replaces all current representations of the target resource with the request payload
DELETE The DELETE method deletes the specified resource.

When you think about it, there isnt a good reason for HTML to not support these extended verbs, We the people say NO MORE ! We want the these verbs to be used by the hypermedia itself, connecting the verbs directly to HTML elements.

This may seem a bit foreign the first time you encounter this, how can you have this kind of power. Who would I want this ? Buckle up because its a bumpy ride.


Motivation
  1. Why should only <a> and <form> be able to make HTTP requests?
  2. Why should only click & submit events trigger them?
  3. Why should only GET & POST methods be available?
  4. Why should you only be able to replace the entire screen? when fetching html ?

By removing these arbitrary constraints, HTMX significantly increases hypertext's potential.

Why does this matter ? Because almost all of the actions you want to do from the web client to the server are http verbs, previous locked up behind javascript and json.
